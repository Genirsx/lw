# 概要设计说明

论文题目：基于区块链的慈善公益管理系统

## 1. 系统总体设计目标

本系统面向慈善公益业务场景，目标是解决传统公益管理过程中项目审核不透明、捐赠和拨付流向难追踪、业务记录可信度不足等问题。系统采用“链下业务管理 + 链上摘要存证”的设计思路，将完整业务数据保存在 MySQL 数据库中，将关键记录摘要写入区块链，实现业务高效处理与数据可信校验的结合。

系统在总体设计上满足以下目标：

- 支持公益项目从申请、审核、上线到募捐、拨付的完整业务闭环。
- 支持捐赠记录、拨付记录和项目关键状态的链上存证。
- 支持普通用户、项目申请者、管理员三类角色的分级使用。
- 支持对历史业务数据进行追溯、校验和审计。
- 支持本地演示、测试链演示和后续扩展部署。

## 2. 系统总体架构

系统采用前后端分离架构，整体由表示层、业务层、数据层和区块链存证层组成。

### 2.1 架构分层

#### 1）表示层

表示层由 React + Vite 构建，负责用户界面展示、表单交互、页面路由和角色工作区切换。前端根据不同用户角色展示不同的模块入口，主要包括游客访问页面、捐赠者工作区、项目申请者工作区和管理员工作区。

#### 2）业务逻辑层

业务逻辑层由 Go 后端实现，负责用户认证、项目管理、审核流程、捐赠处理、拨付处理、链记录查询和日志审计等核心逻辑。后端通过 RESTful 风格接口向前端提供服务，同时封装哈希计算、链交互、权限校验和数据组装逻辑。

#### 3）数据存储层

数据存储层采用 MySQL，负责保存用户信息、项目数据、捐赠记录、拨付记录、链记录和操作日志等结构化数据。数据库作为业务主数据源，支撑系统的查询、统计、审计和权限管理能力。

#### 4）区块链存证层

区块链存证层基于 Solidity 智能合约和以太坊兼容链实现，主要负责接收项目、捐赠和拨付等业务记录的摘要信息，并返回交易哈希、区块号和上链状态。系统不将完整业务数据上链，而是只上链关键摘要，以降低链上存储和计算成本。

### 2.2 总体架构示意

```text
┌──────────────────────────────────────────────┐
│                前端表示层（React）            │
│ 首页 / 登录注册 / 项目大厅 / 管理后台 / 校验页 │
└──────────────────────────────────────────────┘
                    │ HTTP / JSON
                    ▼
┌──────────────────────────────────────────────┐
│               后端业务层（Go API）            │
│ 用户认证 │ 项目管理 │ 审核管理 │ 捐赠管理      │
│ 拨付管理 │ 链记录管理 │ 校验服务 │ 日志审计     │
└──────────────────────────────────────────────┘
          │                           │
          │ SQL                       │ Web3 / RPC
          ▼                           ▼
┌──────────────────────┐   ┌──────────────────────┐
│    MySQL 数据库       │   │ 区块链与智能合约层    │
│ users / projects ... │   │ 项目 / 捐赠 / 拨付摘要 │
└──────────────────────┘   └──────────────────────┘
```

## 3. 系统功能模块设计

根据系统业务目标和角色权限，系统划分为以下核心功能模块。

### 3.1 用户与权限管理模块

该模块负责用户注册、登录、身份识别和访问控制，是系统的基础模块。

主要功能包括：

- 用户注册与登录。
- JWT 身份认证。
- 基于角色的权限控制。
- 当前登录用户信息获取。

模块设计说明：

- 游客可访问首页、项目大厅、链上校验等公共页面。
- 捐赠者可进行捐赠、查看自己的捐赠记录。
- 项目申请者可提交项目申请并查看审核结果。
- 管理员可访问后台管理功能，包括项目审核、拨付、链记录和日志查看。

### 3.2 项目管理模块

该模块负责公益项目的创建、展示、编辑和状态维护，是系统的核心业务模块之一。

主要功能包括：

- 公益项目列表展示。
- 项目详情查看。
- 管理员创建和编辑项目。
- 项目状态切换，如草稿、进行中、已关闭。

模块设计说明：

- 面向公众展示的项目必须经过审核通过。
- 项目包含目标金额、已筹金额、已拨付金额、时间范围、封面图和链状态等信息。
- 项目模块同时作为捐赠模块和拨付模块的数据来源。

### 3.3 项目申请与审核模块

该模块负责项目申请者提交项目、管理员审核项目并决定是否上线。

主要功能包括：

- 项目申请者提交项目申请。
- 管理员查看待审核项目列表。
- 管理员对项目执行通过或驳回操作。
- 保存审核意见、审核人和审核时间。

模块设计说明：

- 项目申请采用“先申请、后审核、再上线”的流程。
- 审核通过后，项目进入公开展示状态。
- 审核驳回后，申请者可查看驳回意见并重新整理材料。

### 3.4 捐赠管理模块

该模块负责捐赠业务的数据采集、存储和展示。

主要功能包括：

- 用户选择项目并发起捐赠。
- 记录捐赠金额、留言、匿名状态等信息。
- 展示项目捐赠明细。
- 展示当前用户的个人捐赠记录。

模块设计说明：

- 捐赠记录生成后会同步触发摘要计算和上链流程。
- 捐赠成功后，系统自动更新对应项目的已筹金额。
- 用户可从“我的捐赠”中查看记录并跳转到链上校验。

### 3.5 资金拨付管理模块

该模块负责管理员对公益项目进行拨付登记与追踪。

主要功能包括：

- 管理员登记拨付记录。
- 记录接收方、用途、说明、拨付时间和金额。
- 展示某项目的拨付列表。
- 查看拨付记录对应的链状态。

模块设计说明：

- 拨付属于后台管理动作，仅管理员可执行。
- 拨付记录同样需要生成摘要并写入区块链。
- 拨付成功后，系统自动更新项目的已拨付金额。

### 3.6 链上存证与校验模块

该模块是系统区别于传统公益管理平台的关键模块，用于实现业务记录可信校验。

主要功能包括：

- 对项目、捐赠、拨付数据生成规范化摘要。
- 调用智能合约写入链上。
- 保存交易哈希、区块号、链状态等回执信息。
- 根据记录编号重新计算哈希并与链上值比对。

模块设计说明：

- 链上仅保存摘要，不直接保存全部业务字段。
- 校验流程采用“数据库哈希”和“链上哈希”双重比对机制。
- 系统支持链记录重试和交易详情查看。

### 3.7 链记录与交易详情模块

该模块用于面向管理员和普通用户展示链上存证结果。

主要功能包括：

- 按业务类型查看链记录。
- 按交易哈希查看交易详情。
- 查看区块高度、交易状态、关联业务和链上原始日志。
- 对失败记录执行重试上链。

模块设计说明：

- 链记录管理有助于系统演示“业务数据如何映射到链上结果”。
- 交易详情页可用于答辩展示系统的可追溯性与可信性。

### 3.8 操作日志与审计模块

该模块用于记录管理员在后台执行的重要操作，形成审计链条。

主要功能包括：

- 记录项目审核、项目发布、拨付登记等关键操作。
- 保存操作者、业务对象、操作时间和附加详情。
- 提供后台分页查询能力。

模块设计说明：

- 该模块强化系统管理闭环，适合论文中强调“可追踪、可审计”的设计目标。

## 4. 系统模块关系设计

各模块之间并不是孤立存在，而是围绕公益项目这一核心业务对象形成协同关系。

### 4.1 核心关系

- 用户与权限管理模块为所有业务模块提供身份识别和权限边界。
- 项目管理模块是捐赠模块、拨付模块和审核模块的业务基础。
- 项目申请与审核模块控制项目是否能够进入公开募捐阶段。
- 捐赠管理模块和资金拨付管理模块共同构成项目资金流转的主体。
- 链上存证与校验模块为项目、捐赠和拨付模块提供可信证明能力。
- 操作日志模块对后台业务动作进行补充记录，形成审计支持。

### 4.2 主要业务流

#### 1）项目申请与上线流

项目申请者提交项目申请后，数据首先进入项目表中的待审核状态；管理员在后台审核通过后，项目变为已通过并公开展示，同时记录审核信息。

#### 2）捐赠业务流

用户提交捐赠后，系统先写入数据库，再生成业务摘要，调用区块链接口写入链上，最后回写交易哈希和链状态，并更新项目筹集金额。

#### 3）拨付业务流

管理员登记拨付后，系统写入拨付表，生成摘要并执行上链，同时回写拨付记录的链信息，并更新项目已拨付金额。

#### 4）校验业务流

用户输入记录编号后，系统读取数据库中的业务记录，重新计算摘要，再与链上摘要进行比较，最终返回校验结果。

## 5. 数据库概要设计

系统数据库采用 MySQL，围绕用户、项目、捐赠、拨付、链记录和日志六类核心对象建模。

### 5.1 主要数据表

#### 1）用户表 `users`

用于保存系统用户的基本信息和角色信息。

核心字段包括：

- 用户编号
- 用户名
- 邮箱
- 密码哈希
- 角色
- 创建时间

#### 2）项目表 `projects`

用于保存公益项目基础信息、审核信息和链状态信息。

核心字段包括：

- 项目编号
- 项目名称
- 项目描述
- 目标金额、已筹金额、已拨付金额
- 项目开始和结束时间
- 项目状态
- 创建者编号
- 审核状态、审核时间、审核人、审核意见
- 链哈希、链状态、交易哈希、区块号

#### 3）捐赠表 `donations`

用于保存用户对项目的捐赠信息。

核心字段包括：

- 捐赠编号
- 关联项目编号
- 捐赠用户编号
- 捐赠人名称
- 是否匿名
- 捐赠金额
- 留言
- 捐赠时间
- 记录哈希、链状态、交易哈希、区块号

#### 4）拨付表 `disbursements`

用于保存项目拨付记录。

核心字段包括：

- 拨付编号
- 关联项目编号
- 拨付金额
- 接收方
- 用途
- 描述说明
- 拨付时间
- 记录哈希、链状态、交易哈希、区块号

#### 5）链记录表 `chain_records`

用于统一保存各类业务对象的上链记录。

核心字段包括：

- 记录编号
- 业务类型
- 业务编号
- 哈希摘要
- 交易哈希
- 区块号
- 上链状态
- 原始载荷

#### 6）操作日志表 `operation_logs`

用于保存后台关键操作的审计记录。

核心字段包括：

- 日志编号
- 用户编号
- 用户名
- 动作名称
- 业务类型
- 业务编号
- 详情信息
- 创建时间

## 6. 接口概要设计

后端采用 REST 风格接口设计，接口可以按业务划分为以下几类：

- 认证接口：注册、登录、获取当前用户。
- 项目接口：项目列表、项目详情、我的项目申请、后台项目管理、项目审核。
- 捐赠接口：捐赠创建、捐赠列表、我的捐赠、捐赠校验。
- 拨付接口：拨付创建、拨付列表、拨付校验。
- 链接口：链摘要、链记录、交易详情、失败重试。
- 日志接口：后台操作日志查询。

该接口划分方式使前端页面与后端服务之间形成清晰的一一映射关系，便于后续维护和扩展。

## 7. 部署与运行设计

系统在部署上采用轻量化结构，适合本地开发、演示和后续扩展。

### 7.1 开发部署结构

- 前端应用部署在 Vite 开发服务器。
- Go 后端提供 HTTP API 服务。
- MySQL 负责业务数据持久化。
- Hardhat 本地链或测试链负责智能合约调用和链上存证。

### 7.2 运行方式说明

- 当前系统支持使用 `mock` 模式进行纯业务演示。
- 在需要展示完整区块链交互时，可切换到本地链或测试链。
- 通过环境变量切换数据库、链模式和合约地址等配置。

## 8. 概要设计特点总结

本系统在概要设计上的主要特点如下：

- 采用前后端分离结构，职责边界清晰。
- 采用分层设计，便于维护和扩展。
- 采用基于角色的功能模块组织方式，贴合真实公益管理流程。
- 采用“链下存业务、链上存摘要”的架构，平衡效率与可信性。
- 采用项目、捐赠、拨付、审核、日志联动设计，形成完整管理闭环。
