# 系统说明与答辩梳理

## 1. 系统定位

本系统用于解决传统慈善场景中“记录可信度不足、流向不透明、审核过程难追踪”的问题。核心思路是把业务数据保存在链下数据库，同时将关键记录摘要写入区块链，实现“链下高效管理，链上可信存证”。

## 2. 技术栈

- 前端：React + Vite
- 后端：Go
- 数据库：SQLite
- 智能合约：Solidity
- 本地链：Hardhat

## 3. 角色划分

### 管理员

- 审核项目申请
- 管理项目上线状态
- 登记拨付
- 查看链记录和操作日志

### 捐赠者

- 浏览已上线项目
- 发起捐赠
- 查看自己的捐赠记录
- 发起记录校验

### 项目申请者

- 提交项目申请
- 查看审核状态
- 查看驳回说明或通过结果

## 4. 核心业务流程

### 项目申请与审核

1. 项目申请者提交项目资料。
2. 项目进入待审核状态。
3. 管理员审核通过后，项目进入公开列表。
4. 若驳回，系统保留审核意见供申请者查看。

### 捐赠上链存证

1. 捐赠者提交捐赠信息。
2. 后端根据业务字段计算规范化哈希。
3. 哈希摘要写入合约。
4. 系统记录交易哈希、区块号、链状态。

### 拨付追踪

1. 管理员登记拨付记录。
2. 拨付记录生成摘要并写入链上。
3. 用户可通过页面查看拨付信息与链上状态。

## 5. 为什么采用链下业务数据 + 链上摘要

- 完整业务数据放数据库中，方便检索和管理
- 链上只保存摘要，降低 gas 成本
- 校验时可重新计算摘要并与链上值比对
- 兼顾性能、成本与可信性

## 6. 前端页面结构

当前前端按角色组织，不再把所有功能堆在同一页：

- 游客：首页、项目大厅、链记录、链上校验
- 捐赠者：首页、我的捐赠、去捐赠、链上校验
- 项目申请者：我的项目申请、已上线项目、链记录
- 管理员：后台首页、项目审核、资金拨付、链记录管理、操作日志

整体使用左侧导航 + 右侧工作区布局，更适合系统型产品展示。

## 7. 适合论文描述的技术点

- 业务数据规范化哈希生成
- 链下数据与链上摘要的一致性校验
- 智能合约对项目、捐赠、拨付三类记录的存证支持
- 基于角色的权限控制
- 管理员审核上线机制
- 本地链和测试链两种演示模式切换

## 8. 演示时可强调的亮点

- 项目不是直接上线，而是要先经过管理员审核
- 捐赠记录和拨付记录都能生成哈希并写入链上
- 支持按交易哈希查看详情
- 支持从前台页面直接校验记录可信性
- 支持查看操作日志，形成完整管理闭环
